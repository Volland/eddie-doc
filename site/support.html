Get help with Eddie Doc

Found a bug, hit a wall, or want a feature? Everything runs through the GitHub issue tracker so it stays public and searchable. Most questions are answered by the troubleshooting notes below.

Where to get help

◈ Bug reports

Open a GitHub issue. Include your VS Code version, the extension version, and — if you can — a small .adoc + annotated PDF that reproduces it.

✦ Feature requests

Describe the workflow you want in a feature issue. Real editorial pain points are the best input.

▤ Questions & docs

Check the Review Format spec and the README first. Still stuck? Open a question issue.

Before you file a bug

A good report gets a fast fix. Please include:

  1. The extension version (Extensions view → Eddie Doc) and your VS Code version (Help → About).
  2. What you did, what you expected, and what happened instead.
  3. If mapping is wrong: a minimal .adoc and the annotated PDF, or the relevant slice of the <file>.review.json sidecar.
  4. Any errors from Help → Toggle Developer Tools → Console.

Troubleshooting

The most common issues, and how to resolve them.

My annotations land under “Unmatched”

The fuzzy matcher couldn't place the marked text above the confidence threshold. Options, in order:

  • Put your cursor on the right line and run Eddie Doc: Re-link to Current Cursor Line, or use the Triage button on the Unmatched group to clear them in one pass.
  • Lower eddieDoc.matchThreshold (default 0.55) if many correct matches are being rejected.
  • If the source and PDF have diverged a lot, enable the semantic fallback (below).

“Semantic fallback couldn't reach Ollama”

The optional embedding fallback is on but no Ollama server answered. Either start Ollama with the model pulled (ollama pull embeddinggemma), point eddieDoc.ollamaUrl at the right host, or turn the feature off with eddieDoc.semanticFallback: false. It is off by default.

Nothing is extracted from my PDF

Eddie Doc reads real PDF markup annotations (highlights, strikethroughs, comments, carets). If your PDF was flattened or printed-to-PDF, the marks are baked into the page image and can't be recovered — re-export keeping annotations live. Reply annotations and empty marks with no comment are skipped by design.

A delete/replace changed more (or less) text than the mark

Character-precise edits use markedText aligned back through AsciiDoc markup. If a mark only partially covered a word, Eddie Doc snaps to whole words. When in doubt, use Replace struck / highlighted text… to review the exact range before applying.

My resolved state disappeared after re-exporting the PDF

Item ids are derived from annotation geometry, so re-exporting a PDF at different positions can produce new ids. Re-map, then re-link any items that lost their state. The <file>.review.json sidecar is a diffable record — commit it to keep review history.

The sidecar looks different after this version

Eddie Doc migrated it to Review Format v2 automatically on the first save. Version-1 sidecars are read transparently; no action needed. See the changelog.
